　HTML / CSS / JavaScript を組み合わせ、<br>
　DBとの連携を意識して実装しました。<br>

## 工夫した点<br>
・ユーザー登録時に、<br>
　ACCOUNTSテーブルにフォームからの入力したものに加え、テーブル名を自動付与。<br>
　そのテーブル名をもとに、テーブルを作成。<br>
　メモテーブルを一人ずつ管理することで、テスト時に確認しやすくしました。<br>

・メモタイトルまたはメモの「検索」機能をつけ、ユーザーが目的のメモを探しやすいようにしました。<br>

・アカウント削除機能をつけ、ユーザーの個人情報の観点にも考慮しました。<br>
　ACCOUNTSテーブルの該当部分・MEMOSテーブル削除することで、関連情報はすべて削除されます。<br>

## 苦労や学び<br>
・「掲示板アプリ」の制作時に、画面に処理が多くなっており、<br>
　一つの画面で分岐して処理・画面表示をするようにしていたため、<br>
　「一つの処理に一つの画面」へ改善しました。<br>
　自分の中で「見た目は同じ画面だから、同じJSPで作成した方がいい」と思っていたところがあり、<br>
　それがかえってコードの把握が難しくなっていたのだと理解しました。<br>
　ユーザーからしてみれば「処理がどう行われようと影響もない」ので、<br>
　役割・状況によって分けることも大事なのだと痛感しました。<br>

・DAOを処理ごとに分けたり、<br>
　途中で「Logic(Model)を通さずにServletからDAOを直接呼び出す」ことを検証しました。<br>
　今回はLogic(Model)の処理が少なかったのですが、<br>
　今後処理が増えることを想定すると、Logic(Model)の必要性を理解できました。<br>

・「1人ずつメモテーブルを作成」するようにしたので、<br>
　アカウント削除時に、「該当者のメモが削除されているか」の確認が容易にできました。<br>
　しかし、「１つの全体メモテーブル」としては管理していなかったため、<br>
　メモテーブル全体としての処理はできない、というのが欠点かなと思っております。<br>

・実装していくと「こういう機能も追加したい」という気持ちが芽生え、<br>
　機能が増えていくのことが多かったのですが、<br>
　今回は「ここまでで終わり」という終了ラインを設けたので、<br>
　スッキリと終われました。そのため、「やりきった」という実感がありました。<br>
